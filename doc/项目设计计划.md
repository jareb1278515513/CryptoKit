## Plan: CryptoKit 期中作业项目设计

采用“Python函数API + CLI”双入口架构，优先完成课程要求的全算法覆盖与可测性。基于你确认的技术路线：ECC 使用 NIST P-160，RC6 使用纯 Python 手写实现。整体按“基础框架 -> 算法模块 -> 接口集成 -> 文档与报告”分阶段推进，确保每阶段可独立验收。

**Architecture（分层架构与路径规范）**
为保证“架构清晰、分层明确、文件分层存放”，项目采用五层结构，并强制按路径隔离职责：

```text
CryptoKit/
├─ main.py                                # 进程入口，仅负责启动 CLI
├─ pyproject.toml
├─ doc/
├─ cryptokit/
│  ├─ interfaces/                         # 接口层（与外部交互）
│  │  ├─ cli/                             # CLI 命令与参数解析
│  │  └─ api/                             # Python API 暴露与适配
│  ├─ application/                        # 应用层（用例编排）
│  │  ├─ use_cases/                       # encrypt/decrypt/sign/verify 等用例
│  │  └─ dto/                             # 输入输出 DTO
│  ├─ domain/                             # 领域层（核心算法与规则）
│  │  ├─ symmetric/                       # AES/SM4/RC6
│  │  ├─ hash/                            # SHA/HMAC/PBKDF2/RIPEMD160
│  │  ├─ encoding/                        # Base64/UTF-8
│  │  └─ asymmetric/                      # RSA/ECC/ECDSA
│  ├─ infrastructure/                     # 基础设施层（第三方库与系统适配）
│  │  ├─ backends/                        # cryptography/gmssl/pycryptodome 封装
│  │  ├─ key_io/                          # 密钥文件读写与序列化
│  │  └─ random/                          # 随机源与熵接口
│  └─ shared/                             # 共享层（跨层公共能力）
│     ├─ errors.py                        # 错误码与异常基类
│     ├─ result.py                        # 统一返回结构
│     └─ types.py                         # 公共类型定义
├─ tests/
│  ├─ unit/                               # 单元测试（按层镜像）
│  ├─ integration/                        # 集成测试（跨层）
│  └─ e2e/                                # CLI/API 端到端测试
└─ examples/
```

分层依赖方向必须单向：`interfaces -> application -> domain`，`infrastructure` 仅通过接口被 `application/domain` 使用，禁止反向依赖。

**Steps**
1. 阶段一：需求落地与骨架设计
- 先创建并锁定分层目录骨架（interfaces/application/domain/infrastructure/shared + tests 分层），确保后续文件按层归档。
- 将作业要求拆解为可验证能力清单：AES/SM4/RC6、SHA族+RIPEMD160+HMAC+PBKDF2、Base64/UTF-8、RSA-1024+RSA-SHA1+ECC-160+ECDSA。
- 在项目层定义统一输入输出协议：成功结果、错误信息、状态码三元结构。
- 规划模块边界：对称加密、哈希与KDF、编码、公钥算法、公共工具、统一 API 导出、CLI 命令分发。

2. 阶段二：工程基础与依赖
- 更新依赖策略：核心库以 cryptography 为主；SM4 与 RIPEMD160 兼容能力按可用性提供适配层；RC6 按纯 Python 自研实现。
- 统一编码与二进制处理规范：字符串默认 UTF-8，算法输入统一 bytes 化，输出支持 hex/base64/raw 三种表示。
- 建立错误码字典与异常映射规则，确保 CLI 与 API 返回一致。*依赖步骤 1*

3. 阶段三：算法实现规划（可并行）
- 子阶段 3A（并行）：编码与哈希模块
  - Base64 编解码、UTF-8 编解码与错误处理。
  - SHA1/SHA256/SHA3、HmacSHA1/HmacSHA256、PBKDF2。
  - RIPEMD160 采用“优先标准库可用实现，失败则回退实现”策略。
- 子阶段 3B（并行）：对称加密模块
  - AES：至少支持常见模式（建议 ECB/CBC/CTR），明确 IV 与填充规则。
  - SM4：按课程要求实现基础加解密能力与模式参数。
  - RC6：纯 Python 手写实现，固定并文档化参数（字长、轮数、分组大小）并提供标准测试向量校验。
- 子阶段 3C（并行）：公钥模块
  - RSA-1024：密钥生成、加解密、RSA-SHA1 签名验签。
  - ECC-160（NIST P-160）：密钥生成与 ECDSA 签名验签。
  - 明确签名输入摘要与编码格式，避免跨模块不一致。

4. 阶段四：双入口集成
- 设计统一 Python API：函数命名、参数规范、返回结构对齐。
- 设计 CLI：按“算法 + 操作”命令结构暴露能力，支持必要参数（key、iv、mode、encoding）。
- 增加接口可调用示例：第三方程序通过 import API 与命令行两种方式调用。*依赖步骤 2、3*

5. 阶段五：测试与验收
- 建立分层测试：模块单测、跨模块一致性测试、CLI 冒烟测试。
- 每类算法至少准备“正确样例 + 异常样例 + 边界样例”。
- 增加已知测试向量校验：重点覆盖 RC6、SM4、RSA/ECDSA 签名验签流程。*依赖步骤 3、4*

6. 阶段六：课程交付材料
- 补充项目文档：系统设计、模块职责、接口契约、错误码定义、使用示例。
- 汇总执行结果：关键命令、输入输出、状态码、异常场景。
- 形成报告内容骨架，对齐评分项五部分（系统设计、代码开发、执行结果、接口调用、参考文献）。*依赖步骤 5*

**Relevant files**
- `main.py` — 作为 CLI 命令分发主入口，落地“算法 + 操作”参数模型。
- `pyproject.toml` — 定义运行依赖、开发测试依赖与入口脚本配置。
- `README.md` — 作为系统设计与使用说明主文档，沉淀接口契约与示例。
- `doc/网络信息安全密码算法编程-期中作业-要求及评分细则.txt` — 需求基线与验收对照来源。

**Verification**
1. 需求覆盖核对：逐项映射四大技术范围，确保每个算法在“API 可调用 + CLI 可执行 + 有测试”三项均满足。
2. 一致性检查：同一算法通过 API 与 CLI 执行得到一致输出（或等价输出编码）。
3. 异常路径检查：错误输入（空值、非法 key/iv、模式不支持）返回预期状态码与可读错误。
4. 可演示性检查：执行流程可见、结果可复现、命令可直接运行，满足课程“可见及可测”要求。
5. 报告对齐检查：五项评分内容均有对应章节、示例与证据截图来源。

**Decisions**
- 已确认：ECC-160 采用 NIST P-160。
- 已确认：RC6 采用纯 Python 手写实现，优先教学可解释性。
- 已确认：接口形态采用“Python API + CLI”，不纳入 HTTP 服务层。
- 已确认：本轮交付目标是“完整可执行开发计划”，非直接进入编码实现。
- 范围内：课程要求所列算法能力、接口可调用、执行可测与文档支撑。
- 范围外：生产级安全加固、分布式部署、Web 管理后台。

**Further Considerations**
1. SM4 依赖策略建议固定为“主实现 + 备用实现”并在文档中写明环境要求，避免评审机器上因依赖差异导致不可运行。
2. RSA-SHA1 与 ECC-160 需在文档中注明“教学用途”，减少与现代安全实践冲突的审阅风险。
